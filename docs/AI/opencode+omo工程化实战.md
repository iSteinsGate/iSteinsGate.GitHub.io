# OpenCode + OmO + Skills：给后端工程师的 AI Agent 工程化实战笔记

## 一、前言
*   **身份**: 拥有15年企业级后端开发经验的技术专家。
*   **核心议题**: 如何将大模型（AI）从一个临时的、基于提示词（Prompt）的聊天工具，**工程化**为“可复用、可审计、可协作、可回归的生产力系统”。

## 二、核心组合：OpenCode + OmO + Skills
这是作者实践验证的一套高性价比工程组合。

| 组件 | 类比 | 核心职责 | 解决的问题 |
| :--- | :--- | :--- | :--- |
| **OpenCode** | 工位 + 工具箱 | 让Agent能在真实代码仓库中工作 | 从“会聊代码”到“能交付代码”（读、改、跑、提交） |
| **OmO** | 项目小组 | 从单Agent升级为可协作的智能体小队 | 将研发中耗时的“拆任务、并行、收敛”流程自动化 |
| **Skills** | 作业指导书/团队SOP | 将高频、规范的流程固化、模板化 | 流程复用、输出风格稳定、风险可控、降低审核成本 |

**三者叠加的转变**: 从“写 Prompt 试试运气”，升级为“按照工程流程交付可审计的成果”。

## 三、OpenCode：在真实项目地图中工作
**核心理念**：为AI Agent配备清晰的“项目地图”，明确边界和规则，这比追求更大模型参数更能提升稳定性。

**实践第一步**：在代码仓库中创建 `AGENTS.md` 等说明文件，包含：
*   **项目结构** (如 `/api`， `/service`)
*   **常用命令** (如 `make run`， `make test`)
*   **变更约束** (如不允许改动哪些目录，文档更新要求)
*   **完成定义(DoD)** (如单测通过、文档同步、变更说明可审计)

## 四、OmO：从单兵到小队
OmO 实现了**多智能体协作**，模拟真实研发团队分工，核心是**并行推进，统一收敛**。

**常用角色划分**:
*   **Planner**: 解读需求规格（Spec），分解任务，识别风险。
*   **Implementer**: 实现核心代码变更。
*   **Tester**: 补充测试用例，进行回归验证。
*   **Doc Writer**: 更新相关API文档、变更说明、决策记录。
*   **Reviewer**: 进行边界检查与安全审查。

**协作关键**: 每个角色必须有明确的“交接产物”，例如：
*   Planner输出：任务清单 + 验收标准
*   Implementer输出：代码变更 + 影响范围
*   Tester输出：测试结果 + 风险提示

## 五、Skills：固化团队SOP
Skills 旨在解决“不能每次都重新教Agent”的问题，将团队的标准操作流程（SOP）以结构化、可复用的方式写入仓库。

**一个Skill示例 (`api-contract`)**:
1.  **输入**: `SPEC.md` 和现有API文档。
2.  **任务**:
    *   从Spec中提取输入输出与约束。
    *   生成接口说明文档。
    *   生成示例请求/响应。
    *   检查幂等、鉴权等非功能性要求。
3.  **输出**: 更新的API文档、错误码表、变更摘要。
4.  **验收标准**: 必须包含错误码表和JSON示例。

**固化后的收益**: 新人与Agent遵循同一流程，输出风格稳定，审核成本低，质量波动小。

## 六、工程流程范式：Spec + 文档驱动 + 严格文档 + Eval
作者认为，在企业研发中，一套规范的**工程流程**比单纯更换模型更能稳定提升交付质量。

1.  **Spec**: 将需求压缩成“工程合同”，明确目标、输入输出、验收标准，让Agent“少靠猜，多靠约束”。
2.  **文档驱动开发**: 建议的最小文档集合包括：
    *   `AGENTS.md`: 项目地图与执行边界
    *   `SPEC.md`: 需求合同
    *   `DECISIONS.md`: 关键设计决策记录
    *   `CHANGELOG.md`: 可追溯的变更记录
3.  **严格文档原则**:
    *   代码变更必须伴随文档变更。
    *   文档必须可追溯（谁、何时、为何修改）。
    *   文档必须可验证（能对应到测试用例和验收标准）。
4.  **Eval回归机制**: 建立一个最小的Eval测试集，每次修改Prompt、Skill或流程后都应运行回归测试，确保质量基线，将改进从“玄学”变为“工程”。

## 七、EvoMap：Agent的“进化层”
EvoMap 旨在解决“重复踩坑、经验难沉淀、缺乏审计链路”的问题，是能力的“沉淀与持续进化层”。

| 对比 | MCP | Skills | EvoMap |
| :--- | :--- | :--- | :--- |
| **比喻** | AI的“肢体”（手和脚） | AI的“招式” | AI的“DNA”（遗传智慧） |
| **解决问题** | 如何连接外部工具和数据源 | 如何执行特定任务 | 能力如何跨个体传承和进化 |
| **核心** | 工具调用标准化 | 任务执行标准化 | 能力遗传、自然选择、经济激励 |
| **经验复用** | ❌ 不涉及 | ❌ 不涉及 | ✅ 核心功能 |
| **跨Agent传播**| ❌ 不涉及 | 有限 | ✅ 全球网络级传播 |

**核心概念**:
*   **Gene**: 能力模板
*   **Capsule**: 经过验证的解决方案
*   **EvolutionEvent**: 进化记录
*   **GDI**: 能力评分指标

## 八、可执行实践流程（以“积分查询接口”为例）
1.  **写Spec**：明确功能、输入输出、验收标准。
2.  **Planner拆任务**：分解子任务，定义依赖。
3.  **Implementer改代码**：进行最小可行改动。
4.  **Tester补测试**：补充单测，进行回归。
5.  **Doc Writer更新文档**：同步API、变更说明、风险。
6.  **跑Eval**：运行回归测试，验证未破坏已有功能。
7.  **生成交付摘要**：提交最终、可审计的变更说明。

**重点**：每一步都有明确产出，整个过程可追溯、可复盘。

## 九、给后端工程师的落地建议
1.  **别先卷Prompt**，先明确项目边界和规则。
2.  **先写Spec**，再让Agent执行。
3.  **建立最小Eval集**，并持续扩充。
4.  **固化高频流程**为Skills。
5.  **所有工具调用都要可审计**。
6.  **权限最小化**，不要给Agent“上帝权限”。
7.  **用Git管理**每一次“自我进化”。
8.  **每次变更必须绑定验收标准**。
9.  **私密信息不要接入**公共上下文。
10. **把AI当同事管理**，而不是当魔法许愿机。

**建议从第一天就跟踪的指标**：
*   交付周期
*   回归缺陷率
*   文档同步率
*   一次验收通过率

## 十、结语
将大模型纳入 **Spec、文档、测试、版本控制、权限管理** 的工程体系后，它就不再是聊天工具，而成为了一个可管理、可复用、可扩展的**生产力系统**。`OpenCode + OmO + Skills + EvoMap` 是一条工程化味道浓厚、值得长期投入的实践路径。
