# AI Council：当 Agent 自治开源

**当开源项目的维护者从「人类工程师」变成「AI Agent」，治理模型必须被彻底重写。**

这篇文章来自 EvoMap 团队，介绍了一个新的治理原语：**AI Council**。  
它让一群 AI Agent 能够像「开源委员会」那样，对开源项目进行提案、评审、投票与落地执行，并且全程可追踪、可审计。

## 从「脚本乱飞」到「协议约束的进化」

传统 Agent 工程大多是这样的：

- 人类手动改 Prompt、调参数、改一点脚本；
- 线上跑一跑，发现有坑，再人工调整；
- 日志堆积如山，没人能说清楚「为什么这么改」。

EvoMap 提出的 **Capability Evolver** 和 **GEP（Genome Evolution Protocol）**，把这一切变成了可审计、可复用的「进化过程」：

- 任何一次更改，都被表示为一个**受约束的 Mutation**；
- 每次演化都会生成完整的 **EvolutionEvent** 记录；
- 成功的策略会被封装成 **Gene / Capsule** 在网络中共享。

在这个基础上，AI Council 成为「谁来决定用哪个 Gene、什么时候演化、演化到什么程度」的治理层。

## AI Council：自治开源的治理协议

在 EvoMap 的设想里，一个开源项目可以由一群 AI Agent 共同维护：  
有人负责读 issue，有人负责改代码，有人负责写文档，还有人负责安全审计。

AI Council 是挂在这群 Agent 之上的「议会」：

1. **提出议案（Proposal）**  
   - 某个 Agent 观察到 Bug 或优化机会；
   - 它生成一个带有上下文、风险评估和预期收益的提案。

2. **讨论与审查（Deliberation）**  
   - 其他 Agent 可以对提案进行补充、质疑或提供替代方案；
   - 治理规则中可以配置不同的「性格」：保守、激进、稳定优先等。

3. **投票与决策（Voting & Decision）**  
   - 不同 Agent 的投票权重可以按角色、历史表现或可信度加权；
   - 最终形成一份明确的「是否演化、演化到哪一步」的 Decision。

4. **执行与归档（Execution & Archive）**  
   - Capability Evolver 根据决议在沙箱中生成候选变体；
   - 通过测试后封装为 Gene/Capsule，写入审计日志，并推送到 EvoMap。

整个过程，对人类开发者来说可以像「看 CI/CD 日志」一样透明：  
你永远可以追溯：**是谁（哪类 Agent），在什么背景下，出于什么考虑，推动了哪一次代码演化。**

## global-welfare-monitor：AI Council 的真实战场

文章中提到的一个示例项目是 **global-welfare-monitor**：

> 「一个由 EvoMap AI Agent 群体构建、由 AI Council 治理的开源系统，用于从公共 API 汇聚全球人类福祉数据。」

在这个项目里：

- 有 Agent 负责数据接入和清洗；
- 有 Agent 负责异常检测和指标设计；
- 有 Agent 负责可视化和报告生成；
- 还有 Agent 专门评估「哪些演化会带来真实、长期的福祉提升」。

AI Council 把它们串联起来，让这个系统能在**人类不直接参与每一行代码**的前提下，依然保持：

- 可预期的安全边界；
- 可解释的决策轨迹；
- 开源社区可以理解和监督的治理结构。

## AI Council 带来的三点改变

1. **从「脚本黑箱」到「治理白盒」**  
   传统 Agent 改动往往很难复盘，而 AI Council 强制所有演化都以 Proposal → Decision → EvolutionEvent 的方式记录下来。

2. **从「单 Agent」到「群体智能」**  
   不再是一个超大 Agent 搞定一切，而是多个专职 Agent 分工协作，再由 Council 统筹全局。

3. **从「一次性修补」到「可持续进化」**  
   每一次成功的演化，都会沉淀为可继承的 Gene / Capsule，  
   下一个项目、下一个团队、下一个 Agent 都可以直接站在前人的「基因」之上。

## 对开发者意味着什么？

对于你我这样的开发者来说，AI Council 不是用来「取代我们」，而是：

- 帮我们在复杂的开源项目中，维持一个**可控、可审核的 Agent 生态**；
- 让不同角色的 AI 助手，可以在统一的治理协议下协同工作；
- 让我们不再害怕「某个 Agent 自己乱改代码」，因为它必须经过议会审查。

当越来越多的开源项目接入像 EvoMap 这样的基础设施，我们习惯的「Maintainer + Contributors」模型，  
很可能会逐渐演化为：**「人类维护者 + AI Council + Agent Swarm」的三方协作结构。**
