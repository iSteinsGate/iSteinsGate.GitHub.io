# OpenClaw x EvoMap：CritPt 评估报告

**当一个以「开放」自居的 Agent 平台，遇上一个以「进化」为核心的基础协议，会发生什么？**

这篇博客围绕 CritPt 物理求解任务，对 **OpenClaw + EvoMap** 的组合进行了系统评估：  
从「语言模仿者」到「物理仿真工程师」，Agent 的能力究竟提升了多少，又暴露了哪些风险？

> 说明：本文为基于公开评估结果的中文解读与梳理，重点在于结构与启示，而非逐句翻译。

## 一、评估任务：CritPt 物理求解

CritPt 是一个用于测试物理推理与仿真能力的基准任务，核心考察：

- Agent 是否理解物理量之间的关系；
- 是否能从自然语言描述中抽象出数学模型；
- 是否能进行稳定的多步推演与误差控制。

评估对象是：

- 仅使用 OpenClaw 作为 Agent 平台；
- 叠加 EvoMap / GEP 能力后的「进化版」 OpenClaw。

目标是看：  
**在「不更换大模型」的前提下，仅通过演化协议与经验继承，系统行为能否发生质变。**

## 二、从「语言模仿者」到「物理工程师」

评估结果表明，裸跑 OpenClaw 时，Agent 在 CritPt 上的表现更像是：

- 能把题目「复述」得有模有样；
- 能写出看起来合理的解释文字；
- 但在关键数值计算与条件边界上，经常出现偏差。

而当引入 EvoMap / GEP 后，系统在几个方面出现了明显提升：

1. **错误模式被系统化记录为 Gene/Capsule**  
   - 每次失败都不再只是一个日志，而是一个待进化的「进化意图」；
   - 修复方案被封装为可复用的物理解题策略。

2. **多轮演化后，Agent 学会了更稳健的求解流程**  
   - 先统一单位与量纲；
   - 再显式写出方程与约束条件；
   - 最后才进行数值求解与结果校验。

3. **新实例可以直接继承老经验**  
   - 当遇到类似类型的物理问题时，Agent 不必再走一遍完整的试错过程；
   - 而是从 Gene 库中拉取「已经在相似环境下验证通过」的解题套路。

简单来说：  
> 通过 GEP，系统把「一次次解题过程」压缩成了可移植的「物理解题基因」。

## 三、安全与鲁棒性：好消息与坏消息

评估并不仅仅关注「能否解出题」，还特别强调了安全与鲁棒性：

- 在**明确、结构化的任务描述**下，系统整体表现稳定；
- 但在**意图模糊、边界不清**或带有**对抗性提示**的场景中，仍然容易做出有风险的操作。

结合社区针对 OpenClaw 的其他安全评估结果，可以看到：

- 在某些安全基准上，OpenClaw 的得分极低（例如安全分仅 2/100 的报告）；
- 暴露出的问题包括：
  - Skills 市场中大量恶意或不受控的能力；
  - 对第三方 MCP 连接缺乏完善的权限与隔离机制；
  - 很容易被提示注入（Prompt Injection）与数据窃取攻击绕过。

引入 EvoMap 后：

- 在「如何解题」这类**能力进化层面**有明显提升；
- 但在「是否应该执行某种高风险操作」这类**安全决策层面**，仍然需要额外的治理与约束。

这提示我们：  
> **进化让能力变强，并不会自动让系统变得更安全。**  
> 安全本身也需要被建模为一种「可验证、可演化的能力基因」。

## 四、协议与平台之间的张力

这篇评估报告实际上也在隐含讨论一个更深层的问题：  
**平台（Platform）和协议（Protocol）之间的角色分工。**

- OpenClaw 更多扮演的是一个「Agent 托管与调用平台」；
- EvoMap / GEP 则更像是「能力如何生长、如何共享」的底层进化协议。

当两者结合时，我们会看到：

- 原本属于「应用层逻辑」的很多经验，被下沉成了可跨项目共享的 Gene；
- 平台本身不再是「唯一的知识容器」，而更像是：
  > 「一块承载不断长出新能力基因的生长基质」。

同时，这也意味着：

- 如果平台层在安全与治理上存在漏洞，那么演化协议可能会加速放大这种风险；
- 反过来，如果协议层把安全与审计内建为「一等公民」，它又可以帮助平台弥补自身短板。

## 五、对未来 Agent 基础设施的启示

通过这次 CritPt 评估，我们大致可以得到几个启示：

1. **光有「会调用工具的 Agent」还不够，需要「会进化的 Agent」。**  
   - 前者在每个任务上都是一次性；  
   - 后者可以让经验在任务之间、项目之间流动。

2. **演化轨迹必须是可审计的。**  
   - 每一次失败、每一次修复，都应该成为后续演化的素材，而不是被日志系统吞掉；
   - GEP 通过 EvolutionEvent / Gene / Capsule 等结构，给出了一个相对完整的表达法。

3. **安全需要被当作「第一类能力」来进化。**  
   - 安全策略也可以被封装为 Gene，在不断对抗攻击与失败中升级；
   - 安全评估结果也应该进入 Agent 的记忆与决策过程。

## 总结

OpenClaw x EvoMap 在 CritPt 评估中的故事，可以简单概括为：

- OpenClaw 提供了一个可以运行 Agent 的场；
- EvoMap / GEP 让这个场里的 Agent **学会了进化和传承**；
- 物理求解能力有了显著提升，但安全与治理仍是必须正面直视的挑战。

如果你在搭自己的 Agent 基础设施，这篇评估报告传达的不是某个产品的胜负，而是一个更底层的判断：

> **未来真正重要的，不是「哪个 Agent 框架更强」，  
> 而是谁能构建出一个既能进化、又能被人类长期信任的智能生态。**
