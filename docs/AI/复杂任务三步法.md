#  复杂任务三步法 (Complex Task Three-Step Methodology) 解析

## 1. 概述
这是一个名为“复杂任务三步法 (S0 → S3)”的 AI 任务处理框架，其核心思想是**分层处理任务复杂度**，旨在以最小的资源消耗，精准地将计算资源（如token）投入到真正复杂的任务中，实现处理效率与质量的最优平衡。

## 2. 核心思想与设计原则
1.  **分层防御 (Defense in Depth)**：采用多级过滤器逐层筛选任务，从零成本的快速判断到深度规划，流量逐层递减，成本逐层递增，以最小化总处理成本。
2.  **灵敏度-成本帕累托平衡**：系统设计倾向于“宁可多评估，也不漏掉复杂任务”，以承受少量误报（简单任务被过度评估）的代价，来避免代价高昂的漏报（复杂任务被当作简单任务执行而失败）。
3.  **规划即铁约束 (Design Freeze)**：深度规划阶段 (S2) 产出的“执行蓝图”是锁定且必须遵守的。任何偏离都需记录原因，确保执行的纪律性，防止无计划的变更导致混乱。
4.  **并行是天然的**：采用 DAG（有向无环图）结构规划任务步骤，支持依赖已满足的步骤自动并行执行，还原真实世界任务的并发性。
5.  **全程质量控制**：质量控制机制（审计、QA、缺陷分级）贯穿执行的每个步骤，确保问题被尽早发现和修复，降低总体返工成本。

## 3. 四层架构详解

### S0：零成本预筛选
*   **目标**：过滤掉绝大多数（约80%）简单请求，零token消耗。
*   **机制**：纯规则匹配。
*   **直接放行 (白名单)**：如简单问答（“几点了”）、延续指令（“继续”）、简单动作（“搜索X”）等。
*   **触发进入S1的信号 (任一命中即触发)**：
    *   **长度**：消息超过200字。
    *   **意图动词**：包含“开发、构建、设计、部署、迁移、重构”等词。
    *   **范围词**：包含“整个、全部、系统、架构、从零开始”等词。
    *   **多步模式**：如“先…然后…最后…”的描述。
    *   **不确定性**：AI无法判断单步执行路径。
    *   **显式触发**：用户提及“复杂任务”、“三步法”等关键词。

### S1：轻量复杂度评估
*   **目标**：对S0筛选出的约20%消息进行快速评分，决定其处理路径。
*   **机制**：消耗约300 token，从五个维度进行1-5分打分。
*   **评估维度**：
    1.  **步骤数**：预估所需步骤。
    2.  **知识域**：涉及的不同领域数量。
    3.  **不确定性**：有多少未知信息需要先行搜索。
    4.  **失败代价**：任务失败后的回退成本。
    5.  **工具链**：需要协调的工具或系统数量。
*   **决策阈值**：
    *   **≤ 8分 (简单)**：直接执行。
    *   **9-15分 (中等)**：进行“轻规划”，列出步骤大纲，边执行边调整。
    *   **> 15分 (复杂)**：进入S2，执行完整的三步法流程。

### S2：深度规划与审计
*   **目标**：为高复杂度任务创建详细、可靠、可并行的执行蓝图，并完成审计锁定。
*   **机制**：采用`Plan Mode`（高能力模型）和`Audit Mode`（审计模型）的循环。
*   **流程**：
    1.  **Plan Mode**：将任务分解为具有依赖关系（`depends_on`）的DAG并行步骤，并标注风险。
    2.  **Audit Mode**：检查规划的完整性、合理性和可行性。
    3.  **修改循环**：进行最多2轮的Plan-Audit修改，直至审计通过。
    4.  **蓝图锁定**：审计通过后，将最终规划锁定为“执行蓝图”，后续执行必须遵循。

### S3：分阶段执行与质量控制
*   **目标**：按锁定的蓝图分阶段、高质量地执行任务，并提供质量控制与问题修复机制。
*   **机制**：分阶段（Phase）执行，同阶段内可并行执行无依赖冲突的步骤。
*   **三道质量防线**：
    1.  **Audit (步骤审计)**：每个步骤完成后立即进行审计。
    2.  **QA (质量保证)**：对步骤产出进行质量检查。
    3.  **Defect Rule (缺陷规则)**：对发现的问题进行分级处理。
*   **成果锁定**：通过QA审计的步骤产出将被“锁定”，供后续阶段使用，确保阶段性成果的稳定性。
*   **缺陷修改四级分级**：
    *   **Critical (严重)**：自动批准修改。
    *   **High (高)**：自动批准修改，并通知人类。
    *   **Medium (中)**：需经人类确认后方可修改。
    *   **Low (低)**：由QA自行决定是否及如何修改。

### 动态升级兜底
在整个流程的任何阶段，如果执行中发现任务实际复杂度被严重低估，可以中途触发重新评估，并升级到完整的S2→S3流程，作为系统安全网。

## 4. 适用范围
该框架不仅限于软件开发，适用于任何被评估为复杂的任务领域，例如：
*   开发（系统构建、架构重构）
*   研究（调研报告、技术选型）
*   运维（故障诊断、系统迁移）
*   内容（长文写作、翻译项目）
*   规划（项目计划、流程设计）

S0的触发规则和S1的评估维度可根据不同领域进行调整配置。

## 5. 与AI代理生态的关系
*   **语义路由**：S0的预筛选复用类似的模式匹配思路。
*   **任务执行规范**：本三步法是更上层的任务执行框架。
*   **Coding Team**：可视为本三步法在编码领域的特化实例。
*   **Heartbeat (主动任务)**：默认经过S0层筛选。

## 6. 关键可配置参数
| 参数 | 默认值 | 说明 |
| :--- | :--- | :--- |
| S0 长度阈值 | 200字 | 触发S1评估的消息长度 |
| S1 简单上限 | 8分 | 低于此分数直接执行 |
| S1 中等上限 | 15分 | 低于此分数进行轻规划，高于则进入S2 |
| S2 最大修改轮数 | 2 | Plan-Audit循环的最大轮数 |
| 动态升级失败阈值 | 2 | 连续失败几次后触发流程升级 |

## 7. 总结
**复杂任务三步法**的核心设计哲学是：**用接近于零的成本区分出需要复杂处理的任务，然后将资源集中、高效、高质量地用于这些任务。** 它通过S0实现高效过滤，通过S2确保复杂任务的规划质量，通过S3保障执行过程的可控与可靠，是一个系统化解决“何时该简、何时该繁”这一工程问题的框架。
